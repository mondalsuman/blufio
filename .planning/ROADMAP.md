## Phase Details

### Phase 53: Data Classification & PII Foundation
**Goal**: Every piece of data in Blufio has a classification level, and PII is automatically detected and redacted across logs, exports, and context
**Depends on**: Nothing (foundation for compliance stack)
**Requirements**: DCLS-01, DCLS-02, DCLS-03, DCLS-04, DCLS-05, PII-01, PII-02, PII-03, PII-04, PII-05
**Success Criteria** (what must be TRUE):
  1. Operator can assign classification levels (Public/Internal/Confidential/Restricted) to memories, messages, and config values via API or CLI
  2. PII patterns (email, phone, SSN, credit card) are detected in content and auto-redacted in log output through the existing RedactingWriter
  3. Data exports redact PII when configured, and PII-containing content auto-classifies as Confidential
  4. Code blocks and URLs are not false-positive flagged as PII
  5. Per-level controls enforce restrictions (Restricted data never exported or sent to LLM, Confidential encrypted at rest and redacted in logs)
**Plans**: 5 plans
Plans:
- [x] 53-01-PLAN.md -- Core types (DataClassification enum, Classifiable trait, PII detection engine, ClassificationGuard)
- [x] 53-02-PLAN.md -- Data layer wiring (DB migration, config, EventBus events, RedactingWriter PII integration)
- [x] 53-03-PLAN.md -- User interface and enforcement (CLI/API endpoints, agent PII detection, context filtering, Prometheus metrics)
- [ ] 53-04-PLAN.md -- Gap closure: Data layer for classification CRUD (Message/Session struct fields, storage queries, StorageAdapter extension)
- [ ] 53-05-PLAN.md -- Gap closure: Wire CLI/API to DB, fix context filtering, add export redaction utility

### Phase 54: Audit Trail
**Goal**: Every security-relevant action in Blufio is recorded in a tamper-evident, hash-chained audit log that can be independently verified
**Depends on**: Phase 53 (classification changes are audit events)
**Requirements**: AUDT-01, AUDT-02, AUDT-03, AUDT-04, AUDT-05, AUDT-06, AUDT-07
**Success Criteria** (what must be TRUE):
  1. Tool executions, memory modifications, config changes, provider calls, session lifecycle events, and classification changes are recorded as hash-chained audit entries
  2. `blufio audit verify` walks the hash chain and reports any tampered or missing entries
  3. Audit trail lives in a dedicated audit.db separate from the main database, with async buffered writes that do not block the agent loop
  4. Audit entries are append-only and exempt from retention deletion, but support GDPR redact-in-place (PII replaced with [ERASED] without breaking hash chain)
**Plans**: 3 plans
Plans:
- [ ] 54-01-PLAN.md -- Core blufio-audit crate (models, chain, writer, filter, migrations, GDPR erasure, config, error types)
- [ ] 54-02-PLAN.md -- BusEvent extension (5 new variants), AuditSubscriber, event emission in memory/agent/gateway
- [ ] 54-03-PLAN.md -- CLI subcommands (verify/tail/stats), serve.rs wiring, doctor health check, backup integration

### Phase 55: Memory Enhancements
**Goal**: Memory retrieval returns the most relevant, diverse, and temporally appropriate results with bounded index size
**Depends on**: Nothing (modifies existing memory crate independently)
**Requirements**: MEME-01, MEME-02, MEME-03, MEME-04, MEME-05, MEME-06
**Success Criteria** (what must be TRUE):
  1. Older memories score lower via configurable temporal decay (default 0.95^days), and explicitly stored memories score higher than extracted ones via importance boost
  2. Memory retrieval results are diverse (MMR reranking eliminates near-duplicate results)
  3. Memory index stays bounded at configurable max entries (default 10,000) with LRU eviction of lowest-scored entries
  4. Background validation detects and flags duplicate, stale, or conflicting memory entries on a configurable schedule
  5. Workspace file changes trigger automatic re-indexing with 500ms debounce
**Plans**: 4 plans
Plans:
- [ ] 55-01-PLAN.md -- Foundation types: MemoryConfig extension, MemorySource::FileWatcher, MemoryEvent::Evicted bulk format, notify deps
- [ ] 55-02-PLAN.md -- Scoring pipeline: temporal decay, importance boost, MMR diversity reranking in retriever.rs
- [ ] 55-03-PLAN.md -- Eviction and validation: store methods, eviction sweep, background validation, combined background task
- [ ] 55-04-PLAN.md -- File watcher, CLI subcommand, Prometheus metrics, serve.rs wiring

### Phase 56: Multi-Level Compaction & Context Budget
**Goal**: Long-running sessions maintain context quality through progressive summarization with quality guarantees, and each context zone enforces its token budget
**Depends on**: Phase 54 (compaction events logged to audit trail)
**Requirements**: COMP-01, COMP-02, COMP-03, COMP-04, COMP-05, COMP-06, CTXE-01, CTXE-02, CTXE-03
**Success Criteria** (what must be TRUE):
  1. Context compaction progresses through 4 levels (L0 raw -> L1 turn-pair summaries -> L2 session summary -> L3 cross-session archive) with soft trigger at 50% context window and hard trigger at 85%
  2. Quality gates evaluate compaction output and reject summaries below threshold (abort at <0.4, retry at 0.4-0.6, proceed at >=0.6) using entity/decision/action/numerical retention scoring
  3. Entity and fact extraction runs before compaction, preserving critical information as separate Memory entries that survive summarization
  4. L3 archive summaries are stored in cold storage with retrieval available for historical context
  5. Zone 1 (static) and Zone 2 (conditional) enforce configurable token budgets with 10% safety margins using accurate provider-specific token counting
**Plans**: 6 plans
Plans:
- [x] 56-01-PLAN.md -- Foundation types: ContextConfig extension, CompactionEvent, delete_messages_by_ids, V13 migration, archive queries
- [x] 56-02-PLAN.md -- Core compaction engine: compaction/ module refactor, L0-L3 levels, dual triggers, cascade, entity extraction
- [x] 56-03-PLAN.md -- Quality scoring and archives: quality gates/retry, L3 archive generation, rolling window, ArchiveConditionalProvider
- [x] 56-04-PLAN.md -- Zone budget enforcement: ZoneBudget struct, static/conditional/dynamic budget enforcement, adaptive budget
- [x] 56-05-PLAN.md -- CLI and wiring: context compact/archive/status commands, serve.rs wiring, Prometheus metrics
- [ ] 56-06-PLAN.md -- Gap closure: Entity persistence to MemoryStore, static zone startup warning, CLI quality score verification

### Phase 57: Prompt Injection Defense
**Goal**: The agent loop is protected against prompt injection attacks at multiple layers without blocking legitimate user input
**Depends on**: Phase 53 (PII detection reused for output validation), Phase 54 (injection attempts logged to audit trail)
**Requirements**: INJC-01, INJC-02, INJC-03, INJC-04, INJC-05, INJC-06
**Success Criteria** (what must be TRUE):
  1. Known injection patterns are detected via regex classifier with 0.0-1.0 confidence scoring, operating in log-not-block mode by default (blocking only at >0.95)
  2. System, user, and external content zones are cryptographically separated via HMAC-SHA256 boundary tokens, preventing cross-zone injection
  3. LLM output is screened for credential leaks and injection relay before tool execution
  4. High-risk operations (tool calls, data export, config changes) trigger configurable human-in-the-loop confirmation
  5. Injection defense applies to MCP client tool output and WASM skill results, not just direct user input
**Plans**: 5 plans
Plans:
- [x] 57-01-PLAN.md -- Crate foundation: blufio-injection crate, L1 pattern classifier, config types, SecurityEvent on EventBus, Prometheus metrics
- [x] 57-02-PLAN.md -- L3 HMAC boundary tokens: per-session key derivation, sign/verify/strip, zone wrapping
- [x] 57-03-PLAN.md -- L4 output screening (credential leak + relay detection) and L5 HITL confirmation manager
- [x] 57-04-PLAN.md -- Integration wiring: pipeline coordinator, agent loop, context engine, MCP/WASM scanning, CLI commands, doctor check
- [ ] 57-05-PLAN.md -- Gap closure: Wire InjectionClassifier to MCP connect_all_with_classifier in serve.rs

### Phase 58: Cron Scheduler & Retention Policies
**Goal**: Blufio runs background tasks on configurable schedules, and data is automatically pruned according to retention rules
**Depends on**: Phase 53 (retention respects classification), Phase 54 (audit entries exempt from retention)
**Requirements**: CRON-01, CRON-02, CRON-03, CRON-04, CRON-05, CRON-06, RETN-01, RETN-02, RETN-03, RETN-04, RETN-05
**Success Criteria** (what must be TRUE):
  1. Operator can define cron jobs in TOML with standard cron expressions, and manage them via `blufio cron list/add/remove/run-now/history`
  2. `blufio cron generate-timers` produces systemd timer unit files for external scheduling
  3. Built-in cron tasks (memory cleanup, backup, cost report, health check, retention enforcement) work out of the box, with job execution history tracked in SQLite
  4. Retention policies with configurable per-type periods (messages, sessions, cost records, memories) automatically enforce via soft-delete with grace period, respecting data classification and exempting audit trail entries
  5. Cron job last-run timestamps persist across process restarts
**Plans**: TBD

### Phase 59: Hook System & Hot Reload
**Goal**: Operators can extend Blufio behavior via shell-based lifecycle hooks, and configuration changes take effect without restart
**Depends on**: Phase 58 (hooks can trigger on cron events), Phase 54 (config changes logged to audit)
**Requirements**: HOOK-01, HOOK-02, HOOK-03, HOOK-04, HOOK-05, HOOK-06, HTRL-01, HTRL-02, HTRL-03, HTRL-04, HTRL-05, HTRL-06
**Success Criteria** (what must be TRUE):
  1. Operator can define shell-based hooks for 11 lifecycle events (pre_start, post_start, pre_shutdown, post_shutdown, session_created, session_closed, pre_compaction, post_compaction, degradation_changed, config_reloaded, memory_extracted) with TOML-defined priority ordering
  2. Hooks receive JSON event context on stdin, execute with configurable timeout and restricted PATH, and a recursion depth counter prevents hook-triggered-hook infinite loops
  3. Editing blufio.toml triggers automatic config reload (parse, validate, ArcSwap swap) with ordered EventBus propagation, and active sessions continue on current config while new sessions use reloaded config
  4. TLS certificates hot-reload via rustls file watcher, and changed WASM skill modules are re-scanned with signature re-verification
  5. config_reloaded lifecycle hook fires after every successful reload
**Plans**: TBD

### Phase 60: GDPR Tooling & Data Export
**Goal**: Operators can fulfill GDPR data subject requests (erasure, portability, transparency) through CLI commands
**Depends on**: Phase 53 (PII detection for export redaction), Phase 54 (erasure logged as audit event), Phase 58 (retention policies for automated enforcement)
**Requirements**: GDPR-01, GDPR-02, GDPR-03, GDPR-04, GDPR-05, GDPR-06
**Success Criteria** (what must be TRUE):
  1. `blufio gdpr erase --user <id>` deletes all user data (messages, memories, session metadata, cost records) and anonymizes cost aggregates
  2. Erasure is logged as an audit trail entry (audit entries themselves are never deleted)
  3. `blufio gdpr report --user <id>` generates a transparency report showing all data held for that user
  4. Data export supports JSON and CSV formats with filtering by session, date range, and data type, with optional PII redaction
  5. Export-before-erasure is configurable as a safety net to prevent accidental permanent data loss
**Plans**: TBD

### Phase 61: Channel Adapters
**Goal**: Blufio can communicate via Email, iMessage, and SMS in addition to existing channels
**Depends on**: Nothing (implements existing ChannelAdapter trait independently)
**Requirements**: CHAN-01, CHAN-02, CHAN-03, CHAN-04, CHAN-05, CHAN-06, CHAN-07
**Success Criteria** (what must be TRUE):
  1. Email adapter polls IMAP for incoming messages and sends replies via SMTP (lettre), with email threads mapped to Blufio sessions via In-Reply-To/References headers
  2. iMessage adapter communicates via BlueBubbles REST API with webhook-based incoming message handling, documented as experimental (requires macOS host)
  3. SMS adapter sends and receives messages via Twilio Programmable Messaging API (webhook inbound, REST outbound)
  4. All three adapters implement ChannelAdapter + PluginAdapter traits and integrate with FormatPipeline

### Phase 62: Observability & API Surface
**Goal**: Operators can trace requests through the system with OpenTelemetry, browse the API with OpenAPI docs, and replicate data with Litestream
**Depends on**: Nothing (independent infrastructure, but benefits from all routes and spans being finalized)
**Requirements**: OTEL-01, OTEL-02, OTEL-03, OTEL-04, OTEL-05, OTEL-06, OAPI-01, OAPI-02, OAPI-03, OAPI-04, LITE-01, LITE-02, LITE-03, LITE-04
**Success Criteria** (what must be TRUE):
  1. OpenTelemetry tracing is available via feature gate with OTLP HTTP export, producing spans for agent loop, LLM calls, tool execution, memory retrieval, and context assembly -- with zero overhead when disabled
  2. Trace context propagates through MCP calls, and OTel coexists with existing Prometheus metrics (OTel for traces, Prometheus for metrics)
  3. OpenAPI 3.1 spec is auto-generated from axum route annotations and served at /openapi.json, with optional Swagger UI at /docs
  4. `blufio litestream init` generates config templates, `blufio litestream status` checks replication lag, and SQLCipher incompatibility is documented with application-level backup as the alternative
**Plans**: TBD

### Phase 63: Code Quality Hardening
**Goal**: The codebase is free of unwrap panics in library code, oversized functions are decomposed, and new subsystems have test coverage
**Depends on**: Phases 53-62 (must sweep all code written in this milestone)
**Requirements**: QUAL-01, QUAL-02, QUAL-03, QUAL-04, QUAL-05, QUAL-06, QUAL-07, QUAL-08
**Success Criteria** (what must be TRUE):
  1. `#![deny(clippy::unwrap_used)]` is enforced across all library crates and all existing unwrap() calls are replaced with proper error handling
  2. `/api/status` returns actual uptime, mock provider uses proper stubs instead of unimplemented!(), and serve.rs and other oversized functions are decomposed
  3. Integration tests exist for channel adapters (Email, iMessage, SMS plus existing adapters)
  4. Property-based tests validate core algorithms (compaction quality scoring, PII detection patterns, hash chain verification)
  5. Benchmark regression detection runs in CI
