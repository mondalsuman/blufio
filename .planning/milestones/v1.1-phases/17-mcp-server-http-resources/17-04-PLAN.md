    2. **handler.rs**: Implement `list_prompts` and `get_prompt`:

       ```rust
       async fn list_prompts(
           &self,
           _request: Option<PaginatedRequestParams>,
           _context: RequestContext<RoleServer>,
       ) -> Result<ListPromptsResult, rmcp::ErrorData> {
           let defs = prompts::list_prompt_definitions();
           let prompts = defs.into_iter().map(|d| {
               rmcp::model::Prompt {
                   name: d.name.into(),
                   description: Some(d.description.into()),
                   arguments: Some(d.arguments.into_iter().map(|a| {
                       rmcp::model::PromptArgument {
                           name: a.name.into(),
                           description: Some(a.description.into()),
                           required: Some(a.required),
                       }
                   }).collect()),
               }
           }).collect();

           Ok(ListPromptsResult { prompts, next_cursor: None, meta: None })
       }

       async fn get_prompt(
           &self,
           request: GetPromptRequestParams,
           _context: RequestContext<RoleServer>,
       ) -> Result<GetPromptResult, rmcp::ErrorData> {
           let args = request.arguments.unwrap_or_default();
           let messages = prompts::get_prompt_messages(&request.name, &args)
               .map_err(|e| rmcp::ErrorData::invalid_params(e, None))?;

           let prompt_messages = messages.into_iter().map(|m| {
               rmcp::model::PromptMessage {
                   role: match m.role.as_str() {
                       "system" => rmcp::model::Role::Assistant, // MCP uses assistant for system
                       "user" => rmcp::model::Role::User,
                       _ => rmcp::model::Role::User,
                   },
                   content: rmcp::model::PromptMessageContent::Text(
                       rmcp::model::TextContent { text: m.content.into(), ..Default::default() }
                   ),
               }
           }).collect();

           Ok(GetPromptResult {
               description: None,
               messages: prompt_messages,
               meta: None,
           })
       }
       ```

       Note: Check exact rmcp types for PromptMessage, PromptMessageContent, Role, TextContent. The exact type structure may differ -- use Context7 or rmcp docs to verify. Adapt field names as needed.

    1. **notifications.rs**: Create `crates/blufio-mcp-server/src/notifications.rs`:

       This module provides the infrastructure for sending MCP notifications. Two notification types:

       **tools/list_changed (SRVR-13):**
       - Emitted when the tool set changes (skill install/discovery)
       - Implementation: The handler tracks the current tool count. When `list_tools` is called and the count differs from the last known value, it means tools changed.
       - Better approach: Provide a `notify_tools_changed` method that external code (serve.rs) can call.

       ```rust
       use std::sync::atomic::{AtomicUsize, Ordering};
       use std::sync::Arc;
       use tokio::sync::watch;

       /// Notification channel for tool list changes.
       /// The sender is held by serve.rs; the receiver is polled by the handler.
       pub fn tools_changed_channel() -> (ToolsChangedSender, ToolsChangedReceiver) {
           let (tx, rx) = watch::channel(0u64); // generation counter
           (ToolsChangedSender(tx), ToolsChangedReceiver(rx))
       }

       pub struct ToolsChangedSender(watch::Sender<u64>);
       impl ToolsChangedSender {
           pub fn notify(&self) {
               let _ = self.0.send_modify(|v| *v += 1);
           }
       }

       pub struct ToolsChangedReceiver(watch::Receiver<u64>);
       impl ToolsChangedReceiver {
           pub fn changed(&mut self) -> impl std::future::Future<Output = Result<(), watch::error::RecvError>> + '_ {
               self.0.changed()
           }
       }
       ```

       Note: The actual emission of the notification via the MCP protocol requires access to the session's `RequestContext::peer`. Since `tools/list_changed` is a server-initiated notification, it needs to be sent through the active MCP session. rmcp's `StreamableHttpService` manages sessions internally.

       For Phase 17, implement the channel and document how serve.rs will connect it. The actual notification emission (sending `notifications/tools/list_changed` via the MCP protocol to connected clients) may require rmcp API investigation. If rmcp provides a `ServerSession` or `Peer` handle for sending notifications, wire it. If not, document the gap.
