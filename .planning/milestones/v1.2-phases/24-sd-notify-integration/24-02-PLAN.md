  <action>
Modify `serve.rs` to call sdnotify functions at three initialization milestones plus spawn the watchdog task. All calls go through `blufio_agent::sdnotify::` (the module created in Plan 01).

1. **STATUS= at vault milestone** -- After the `info!("vault unlocked -- secrets available");` line (inside `Ok(Some(_vault))` branch, line 98), add:
```rust
blufio_agent::sdnotify::notify_status("Initializing: vault unlocked");
```
Do NOT add a STATUS= for the `Ok(None)` (no vault) case -- per user decision, STATUS= is for significant milestones only.

2. **READY=1 at channel connection** -- After the `info!(channels = mux.channel_count(), "channel multiplexer connected");` line (line 501-503), add:
```rust
{
    let ready_status = format!(
        "Ready: {} channel{}{}",
        mux.channel_count(),
        if mux.channel_count() == 1 { "" } else { "s" },
        if memory_provider.is_some() { ", memory enabled" } else { "" }
    );
    blufio_agent::sdnotify::notify_ready(&ready_status);
}
```
This is the critical READY=1 point -- it MUST come after `mux.connect().await?` succeeds and BEFORE the agent loop runs. This satisfies SYSD-01 (ready only after all initialization completes).

Note: `memory_provider` is an `Option<MemoryProvider>` that is in scope at this point. `mux.channel_count()` returns the number of connected channels.

3. **Spawn watchdog task** -- After the memory monitor spawn block (after line 587, after `info!("memory monitor started")`), add a new block:
```rust
// Spawn sd_notify watchdog ping task (if systemd watchdog is enabled).
{
    let wd_cancel = cancel.clone();
    let _watchdog_handle = blufio_agent::sdnotify::spawn_watchdog(wd_cancel);
}
```
This follows the same pattern as the memory monitor spawn. The `_watchdog_handle` is intentionally unused (prefixed with `_`) -- the task runs in the background and is cancelled via the CancellationToken.

4. **Do NOT add any other STATUS= calls.** Per user decision: 3-4 milestones only (vault init, channel connection/ready). The shutdown STATUS= calls are already in shutdown.rs from Plan 01.

5. **Verify no direct sd_notify imports.** The `use` block at the top of serve.rs should NOT import anything from the `sd_notify` crate. All calls go through `blufio_agent::sdnotify::`.

<task type="auto">
  <name>Task 2: Update systemd unit file for Type=notify with watchdog</name>
  <files>contrib/blufio.service</files>
  <action>
Update `contrib/blufio.service` with the following changes to the `[Service]` section:

1. Change `Type=simple` to `Type=notify`.

2. Add `NotifyAccess=main` on the line after `Type=notify`. (This is the default for Type=notify but makes the intent explicit per user decision.)

3. Remove the entire `ExecStartPost` line:
```
ExecStartPost=/bin/sh -c 'for i in $(seq 1 15); do curl -sf http://127.0.0.1:3000/health > /dev/null 2>&1 && exit 0; sleep 1; done; exit 1'
```
This health-check polling loop is redundant with Type=notify -- systemd waits for READY=1 instead.

4. Add `TimeoutStartSec=90` after the `RestartSec=5s` line (or in the timing section). This covers first-run model download which can take 30-60 seconds.

5. Add `WatchdogSec=30` after `TimeoutStartSec=90`. With WatchdogSec=30 and the watchdog pinging at half (15s), systemd restarts Blufio if it hangs for more than 30 seconds.

6. Add a comment above the ExecStart line explaining the Type=notify behavior:
```ini
# Type=notify: systemd waits for READY=1 from sd_notify before marking active.
# Watchdog: blufio pings every WatchdogSec/2; systemd restarts on missed pings.
```

The final `[Service]` section should look like:
```ini
[Service]
Type=notify
NotifyAccess=main
ExecStartPre=-/var/lib/blufio/hooks/pre-start.sh
# Type=notify: systemd waits for READY=1 from sd_notify before marking active.
# Watchdog: blufio pings every WatchdogSec/2; systemd restarts on missed pings.
ExecStart=/usr/local/bin/blufio serve
ExecStopPost=-/var/lib/blufio/hooks/post-stop.sh
Restart=on-failure
RestartSec=5s
StartLimitBurst=3
StartLimitIntervalSec=60
TimeoutStartSec=90
TimeoutStopSec=30
WatchdogSec=30
```

Do NOT change the `[Unit]` section, `[Install]` section, security hardening directives, user/group, environment, or memory limits.
