<objective>
Add PRAGMA integrity_check verification to backup and restore operations with corruption handling, auto-rollback, and operator-facing status reporting.

Purpose: Ensure operators can trust that backups are not silently corrupt and restores produce a valid database. Currently, backup writes a file without verifying it, and restore validates only with `SELECT 1` (proves file opens, not that data is intact). This plan replaces weak validation with full SQLite integrity verification.

Output: Modified `backup.rs` with integrity check helper, verified backup output, pre-check + post-check + rollback restore flow, and comprehensive tests for corruption handling.
</objective>

  <behavior>
    - run_integrity_check on a valid SQLite file returns Ok(())
    - run_integrity_check on a corrupt SQLite file returns Err with "integrity check failed" message containing the first error
    - run_integrity_check on an empty (new) SQLite file returns Ok(())
    - run_backup on a valid source DB produces a backup file that passes integrity check
    - run_backup output (stderr) matches format "Backup complete: X.X MB, integrity: ok"
    - When backup file is corrupt (simulated by post-write corruption), the corrupt file is deleted and error is returned
    - run_backup output on failure includes cause and action: "Backup FAILED: integrity check failed (...). Backup file deleted."
  </behavior>
  <action>
1. Add a new function `run_integrity_check(path: &Path) -> Result<(), BlufioError>` to backup.rs:
   - Open a read-only connection with `SQLITE_OPEN_READ_ONLY | SQLITE_OPEN_NO_MUTEX` flags
   - Execute `PRAGMA integrity_check(1)` (limit to 1 error row per RESEARCH pitfall #3 -- faster on corrupt databases)
   - Collect rows. If single row == "ok", return Ok(())
   - Otherwise, extract first error row and return `Err(BlufioError::Storage { source: Box::new(std::io::Error::new(ErrorKind::InvalidData, format!("integrity check failed ({first_error})"))) })`
   - The function naturally drops the connection when it returns, avoiding pitfall #1 (connection left open during file deletion)

2. Modify `run_backup()`:
   - After `backup.run_to_completion()` succeeds, call `run_integrity_check(Path::new(backup_path))`
   - On integrity check failure: delete the backup file with `let _ = std::fs::remove_file(backup_path)` (ignore delete errors), print failure message to stderr, return the error
   - Failure message format: `eprintln!("Backup FAILED: {e}. Backup file deleted.")`
   - On success: change existing output from `"Backup complete: {size_mb:.1} MB written to {backup_path}"` to `"Backup complete: {size_mb:.1} MB, integrity: ok"`
   - Suggest next step on failure: `eprintln!("Run 'blufio doctor' for full database diagnostics.")`

3. Add tests:
   - `test_integrity_check_valid_db`: Create DB with data, verify run_integrity_check returns Ok
   - `test_integrity_check_empty_db`: Create empty DB, verify run_integrity_check returns Ok
