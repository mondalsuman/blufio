  <name>Task 2: Implement real WASM host functions and use traps for denied capabilities</name>
  <files>crates/blufio-skill/src/sandbox.rs</files>
  <action>
    Fix the WASM host function implementations in `define_host_functions()` to:
    1. Use wasmtime traps (return `Result<i32, wasmtime::Error>`) for denied capabilities instead of returning -1
    2. Implement real http_request, read_file, write_file bodies when the capability IS permitted

    **Capability denial -- use traps:**

    For http_request, read_file, write_file when capability is NOT permitted, change the return type from `-> i32` to `-> Result<i32, wasmtime::Error>` and return:
    ```rust
    Err(anyhow::anyhow!("capability not permitted: skill lacks network/filesystem permission").into())
    ```
    This causes the WASM execution to trap, which is caught by the invoke() error handler and returned as a SkillResult with is_error=true and a descriptive message.

    Note: wasmtime's `func_wrap` accepts closures returning `Result<T, wasmtime::Error>` which produces traps on Err. Use `anyhow::anyhow!` since wasmtime uses anyhow internally.

    **http_request implementation (when permitted):**

    Since wasmtime func_wrap is synchronous and reqwest is async, use `std::thread::spawn` + `tokio::runtime::Handle::current()` to bridge:
    - Actually, for a simpler approach: since the host functions are sync and we can't easily do async HTTP from within func_wrap without async support enabled, implement http_request as a blocking call using `reqwest::blocking::Client`.
    - BUT: the current wasmtime Config does NOT enable async_support. The simpler approach is:
      1. Read URL from WASM memory using the existing `read_string_from_memory` helper
      2. Validate the URL domain against `manifest.capabilities.network.domains` (the domain allowlist)
      3. Use `reqwest::blocking::Client` to make the request (add `reqwest` with `blocking` feature to blufio-skill/Cargo.toml if not already present)
      4. Apply SSRF validation via `blufio_security::ssrf::validate_url_host()`
      5. Write response body back to WASM memory using `write_bytes_to_memory`
      6. Return the response status code as i32 (or -1 on error)

    **IMPORTANT SIMPLIFICATION:** Given the complexity of memory management for HTTP responses in synchronous WASM context (the guest needs to allocate memory for the response), implement a simpler approach:
    - http_request: Read URL from memory, validate domain, make blocking request, store response body in SkillState.result_json (overwriting set_output). Return status code or -1 on error. This way the skill gets the response through the standard output mechanism.
    - Actually, the cleaner pattern is: store the HTTP response in a new field on SkillState (e.g., `last_http_response: Option<String>`), and add a `get_http_response_len` + `get_http_response` pair of host functions for the guest to read it. BUT this adds complexity beyond what the gap closure needs.

    **PRAGMATIC APPROACH for host functions:**
    Given the complexity of full async HTTP/file implementations through WASM memory management, implement the following minimum viable versions:

    For **http_request**: Read URL and method from WASM memory. Validate domain against manifest allowlist. Use `reqwest::blocking::Client` (ensure `reqwest/blocking` feature is available) to make the request. Store the response body in `caller.data_mut().result_json`. Return HTTP status code as i32, or trap if SSRF/domain validation fails.

    For **read_file**: Read path from WASM memory. Validate that the path starts with one of the manifest's `filesystem.read` paths. Use `std::fs::read_to_string`. Store content in `caller.data_mut().result_json`. Return content length as i32, or trap on error.

    For **write_file**: Read path and data from WASM memory. Validate that the path starts with one of the manifest's `filesystem.write` paths. Use `std::fs::write`. Return 0 on success, or trap on error.

    **Changes needed in sandbox.rs:**

    1. Add `use anyhow::anyhow;` at the top (wasmtime already depends on anyhow)

    2. For all three gated host functions, change return type from `-> i32` to `-> Result<i32, wasmtime::Error>` and wrap existing returns in Ok()/Err()

    3. For capability-denied case: `return Err(anyhow!("capability not permitted: ...").into());`

    4. For capability-permitted case: implement actual behavior as described above

    5. Add `allowed_domains` clone for http_request closure, `read_paths` / `write_paths` clones for file closures

    6. Update SkillState to clone manifest capabilities for access inside closures

    **Cargo.toml changes:**
    - Ensure `reqwest` with `blocking` feature is in blufio-skill/Cargo.toml dependencies
    - Ensure `blufio-security` is in blufio-skill/Cargo.toml dependencies (for SSRF validation)

    **Testing:** The existing WAT-based tests for fuel exhaustion and epoch timeout should still pass. Add a test that verifies a capability-denied call produces a trap error (not a -1 return). The test should load a WAT module that calls http_request, with a manifest that has NO network capability, and assert the invoke result contains "capability not permitted".
