  **Step 2: Create heartbeat.rs**

  Design the HeartbeatRunner:

  ```rust
  use std::collections::hash_map::DefaultHasher;
  use std::hash::{Hash, Hasher};
  use std::sync::Arc;

  use blufio_config::model::{CostConfig, HeartbeatConfig};
  use blufio_core::error::BlufioError;
  use blufio_core::types::{ProviderMessage, ProviderRequest, ContentBlock, TokenUsage};
  use blufio_core::{ProviderAdapter, StorageAdapter};
  use blufio_cost::{BudgetTracker, CostLedger};
  use blufio_cost::ledger::{CostRecord, FeatureType};
  use blufio_cost::pricing;
  use tokio::sync::Mutex;
  use tracing::{debug, info, warn};

  /// Result of a heartbeat execution.
  #[derive(Debug, Clone)]
  pub struct HeartbeatResult {
      /// Generated heartbeat content from Haiku.
      pub content: String,
      /// Token usage for cost tracking.
      pub usage: TokenUsage,
      /// Whether the heartbeat had actionable content.
      pub has_content: bool,
  }

  /// Manages periodic proactive check-ins using Haiku.
  ///
  /// Runs on a configurable interval, checks for changes since last heartbeat,
  /// and generates proactive insights when state has changed. Uses a dedicated
  /// budget tracker separate from conversation costs.
  pub struct HeartbeatRunner {
      config: HeartbeatConfig,
      provider: Arc<dyn ProviderAdapter + Send + Sync>,
      storage: Arc<dyn StorageAdapter + Send + Sync>,
      cost_ledger: Arc<CostLedger>,
      /// Dedicated budget tracker for heartbeat costs only.
      budget_tracker: Mutex<BudgetTracker>,
      /// Hash of the state at last heartbeat execution.
      last_state_hash: Mutex<u64>,
      /// Content of the last heartbeat (for skip detection and dedup).
      last_heartbeat_content: Mutex<Option<String>>,
      /// Pending heartbeat content for on_next_message delivery.
      pending_heartbeat: Mutex<Option<String>>,
      /// Count of messages processed since last heartbeat.
      messages_since_last: Mutex<u64>,
  }
  ```

  Implement HeartbeatRunner:

  - `pub fn new(config: HeartbeatConfig, provider: Arc<...>, storage: Arc<...>, cost_ledger: Arc<CostLedger>) -> Self`:
    - Create dedicated BudgetTracker with CostConfig that has `daily_budget_usd: None` and `monthly_budget_usd: Some(config.monthly_budget_usd)`
    - Initialize last_state_hash to 0, pending_heartbeat to None, messages_since_last to 0

  - `pub async fn should_skip(&self) -> bool`:
    - Compute current state hash
    - Compare with last_state_hash
    - If equal, return true (skip)
    - Also skip if budget is exhausted

  - `fn compute_state_hash(&self, message_count: u64, date: &str) -> u64`:
    - Hash together: message_count, date string
    - Use DefaultHasher
    - This is intentionally simple -- we want heartbeats to fire periodically even if "nothing changed" based on just the date changing

  - `pub async fn execute(&self) -> Result<Option<HeartbeatResult>, BlufioError>`:
    1. Check should_skip() -- if true, return Ok(None)
    2. Check budget via dedicated budget_tracker
    3. Get recent sessions and their last few messages for context
    4. Build the heartbeat prompt
    5. Call provider.complete() with Haiku model (non-streaming, shorter response)
    6. Record cost with FeatureType::Heartbeat
    7. Update last_state_hash and messages_since_last
    8. If response is substantive (not "nothing to report"), store as pending or return for immediate delivery
    9. Return HeartbeatResult

  - `fn build_heartbeat_prompt(&self, session_summaries: &[String], current_date: &str) -> ProviderRequest`:
    System prompt:
    ```
    You are a personal assistant performing a periodic check-in. Review the recent conversation context and identify any:
    - Pending items the user mentioned they would do
    - Reminders or follow-ups that might be relevant
    - Time-sensitive items based on today's date

    If there is nothing actionable to report, respond with exactly: "NO_HEARTBEAT"

    If there IS something worth mentioning, write a brief, friendly check-in message (1-3 sentences).
    Prefix your message with "[Check-in] " so the user knows this is proactive, not a response.
    ```
    Build ProviderRequest with:
    - model: self.config.model (Haiku)
    - max_tokens: 256 (short responses)
    - stream: false (use complete, not stream)
    - messages: user message containing session summaries and date

  - `pub async fn notify_message_received(&self)`:
    - Increment messages_since_last counter
    - Called by the agent loop when any message comes in

  - `pub async fn take_pending_heartbeat(&self) -> Option<String>`:
    - Takes and returns the pending heartbeat content (for on_next_message delivery)
    - Returns None if no pending heartbeat

  - `pub async fn record_heartbeat_cost(&self, usage: &TokenUsage) -> Result<(), BlufioError>`:
    - Calculate cost using pricing::get_pricing with heartbeat model
    - Create CostRecord with FeatureType::Heartbeat, session_id = "heartbeat"
    - Record in cost_ledger
    - Record in dedicated budget_tracker
