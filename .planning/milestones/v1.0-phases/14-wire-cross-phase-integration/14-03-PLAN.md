# Plan 14-03: Wire Prometheus Business Metric Call Sites

## Objective

Add runtime call sites for all Prometheus recording functions (`record_message`, `record_tokens`, `record_error`, `set_active_sessions`, `set_budget_remaining`, `record_latency`) in the agent loop and session actor code. Currently these functions exist but are never called at runtime. After this plan, the `/metrics` endpoint will export non-zero business metrics.

## must_haves

- [ ] `blufio_messages_total` counter incremented for every inbound message (labeled by channel)
- [ ] `blufio_tokens_total` counter incremented for every LLM response (labeled by model and type: input/output)
- [ ] `blufio_errors_total` counter incremented for every error (labeled by type: provider/security/storage/agent)
- [ ] `blufio_active_sessions` gauge updated when sessions are created/closed
- [ ] `blufio_budget_remaining_usd` gauge updated after each cost recording
- [ ] `blufio_response_latency_seconds` histogram records LLM API call latency (time from stream request to MessageStop)
- [ ] All metric calls are gated behind `#[cfg(feature = "prometheus")]`
- [ ] Prometheus feature disabled: zero runtime overhead (compile-time elimination)
- [ ] Token metrics use per-model labels for cost distribution visibility
- [ ] Error metrics labeled by error type matching BlufioError variants

## Context

**Current state:** `blufio_prometheus::recording` defines all recording functions (recording.rs). They are registered at startup by `PrometheusAdapter::new()`. The memory monitor in serve.rs already calls `set_memory_heap`, `set_memory_rss`, `set_memory_resident`, `set_memory_pressure` -- but the business metrics (`record_message`, `record_tokens`, `record_error`, `set_active_sessions`, `set_budget_remaining`, `record_latency`) have zero call sites.

**Target state:** Every message processing path records metrics. The `/metrics` endpoint shows live counters for messages, tokens, errors, latency, sessions, and budget.

**Pattern to follow:** The memory monitor in serve.rs already demonstrates the `#[cfg(feature = "prometheus")]` gating pattern. Apply the same pattern to all new call sites.

   a. **After receiving inbound message** (after line ~171, in the `handle_inbound` method, right after the debug log):
   ```rust
   // Record inbound message metric.
   #[cfg(feature = "prometheus")]
   blufio_prometheus::record_message(&channel_name);
   ```

   b. **After session creation in `resolve_or_create_session()`** (after the "created new session" info log, ~line 614):
   ```rust
   #[cfg(feature = "prometheus")]
   blufio_prometheus::set_active_sessions(self.sessions.len() as f64);
   ```

   c. **Also update session count when resuming** (after inserting existing session actor into map, ~line 588):
   ```rust
   #[cfg(feature = "prometheus")]
   blufio_prometheus::set_active_sessions(self.sessions.len() as f64);
   ```

   d. **On error in handle_inbound** (in the error branch of channel.receive, ~line 133):
   ```rust
   #[cfg(feature = "prometheus")]
   blufio_prometheus::record_error("agent");
   ```

   e. **On error from handle_inbound** (in the error branch at ~line 129):
   ```rust
   if let Err(e) = self.handle_inbound(inbound).await {
       error!(error = %e, "failed to handle inbound message");
       #[cfg(feature = "prometheus")]
       {
           let error_type = classify_error_type(&e);
           blufio_prometheus::record_error(error_type);
       }
   }
   ```

   f. **Add error classification helper** at the bottom of lib.rs:
   ```rust
   /// Classify a BlufioError into a Prometheus-compatible error type label.
   #[cfg(feature = "prometheus")]
   fn classify_error_type(error: &BlufioError) -> &'static str {
       match error {
           BlufioError::Provider { .. } => "provider",
           BlufioError::Security(_) => "security",
           BlufioError::Storage { .. } => "storage",
           BlufioError::Config(_) | BlufioError::Internal(_) => "agent",
           BlufioError::BudgetExhausted { .. } => "agent",
           _ => "agent",
       }
   }
   ```

4. **Add LLM response latency tracking** in `handle_inbound()`:

   a. Before calling `actor.handle_message(inbound).await` (~line 213), capture start time:
   ```rust
   let llm_start = std::time::Instant::now();
   ```

   b. After `consume_stream` returns for the first time in the tool loop (~line 254), record latency:
   ```rust
   #[cfg(feature = "prometheus")]
   {
       let latency = llm_start.elapsed().as_secs_f64();
       blufio_prometheus::record_latency(latency);
   }
   ```

   Note: This measures the full round-trip (handle_message + stream consumption), which is the end-to-end latency the user experiences. For pure API latency, we'd measure only the stream consumption, but end-to-end is more useful for operations.
