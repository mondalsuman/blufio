    **Test infrastructure (helper functions):**
    - `setup_encrypted_db()` -> creates a temp file, opens with BLUFIO_DB_KEY env var set, runs migrations, registers sqlite-vec, returns (Connection, temp_dir)
    - `setup_plaintext_db()` -> same but without encryption, for simpler tests
    - `create_synthetic_embedding(seed: u64)` -> generates a deterministic 384-dim f32 vector from seed (normalized, so cosine distance is meaningful). Use a simple deterministic pattern: fill with seed-derived values, then normalize.
    - `create_test_memory(id: &str, content: &str, seed: u64)` -> returns a Memory struct with synthetic embedding

    **Test 1: test_sqlcipher_vec0_compatibility (VEC-02)**
    - Create encrypted database with BLUFIO_DB_KEY
    - Register sqlite-vec via ensure_sqlite_vec_registered()
    - Verify vec_version() succeeds on encrypted connection
    - Create vec0 table (migration V15)
    - INSERT a row into memories_vec0
    - KNN search returns the inserted row
    - This proves vec0 shadow tables work under SQLCipher encryption

    **Test 2: test_vec0_parity_with_in_memory**
    - Create 20 test memories with known synthetic embeddings
    - Save via MemoryStore with vec0_enabled=true (dual-write)
    - Populate vec0
    - Run vector_search via HybridRetriever with vec0_enabled=true
    - Run vector_search via HybridRetriever with vec0_enabled=false (in-memory)
    - Assert: same memory IDs returned (order may differ slightly due to floating point)
    - Assert: similarity scores are within 0.01 of each other

    **Test 3: test_vec0_metadata_filtering_vec03 (VEC-03)**
    - Create 10 memories: 5 active, 2 forgotten, 2 superseded, 1 restricted
    - Save all via MemoryStore with vec0_enabled=true
    - Update status on forgotten/superseded memories (sync to vec0)
    - KNN search should return exactly 5 results (only active, non-restricted)
    - Verify no forgotten, superseded, or restricted memories appear
    - This proves metadata columns filter DURING KNN, not post-query

    **Test 4: test_vec0_fallback_on_failure**
    - Create MemoryStore with vec0_enabled=true but do NOT create vec0 table (skip migration V15)
    - Save memories without vec0 (vec0_enabled=false initially, then construct retriever with vec0_enabled=true)
    - Or: create a scenario where vec0 query fails (e.g., corrupt the vec0 table)
    - Call retrieve() -- should fall back to in-memory search and return results
    - Verify results are returned (not an error)

    **Test 5: test_vec0_dual_write_atomicity**
    - Save a memory with vec0_enabled=true
    - Verify row exists in both memories table and vec0 table
    - Verify rowids match

    **Test 6: test_vec0_eviction_sync**
    - Save 5 memories with vec0_enabled=true
    - batch_evict 2 memories
    - Verify vec0 count decreased by 2
    - KNN search does not return evicted memories

    2. If needed, add test dependencies to crates/blufio/Cargo.toml (e.g., tempfile for temp dirs). Check if tempfile is already a dev-dependency.

    3. Note: Some tests may need to use `#[tokio::test]` with the tokio runtime. Follow existing e2e test patterns (check crates/blufio/tests/e2e.rs for runtime configuration).

    4. For the SQLCipher test, set BLUFIO_DB_KEY env var using `std::env::set_var` in the test setup. Use a test-specific key like "test-vec0-key-12345".
