    **Gap 1 fix -- serve.rs notification channel wiring (SRVR-13):**

    In `crates/blufio/src/serve.rs`, inside the `#[cfg(feature = "mcp-server")] if config.mcp.enabled { ... }` block (around lines 317-353), add the notification channel creation and wiring:

    1. After the line that creates `mcp_handler` (line 334), call `tools_changed_channel()` and wire the receiver:

       ```rust
       // Create tools-changed notification channel.
       let (tools_changed_tx, tools_changed_rx) =
           blufio_mcp_server::notifications::tools_changed_channel();

       let mcp_handler = blufio_mcp_server::BlufioMcpHandler::new(
           tool_registry.clone(),
           &config.mcp,
       )
       .with_resources(
           memory_store.clone(),
           Some(storage.clone() as Arc<dyn blufio_core::StorageAdapter + Send + Sync>),
       )
       .with_notifications(tools_changed_rx);
       ```

    2. After `gateway.set_mcp_router(mcp_router).await;`, store the sender so it remains alive. The sender must be held in a variable that lives until the end of `run_serve`. Use a `let` binding outside the `#[cfg]` block or assign it inside the block and bind to a variable in the outer scope. The simplest approach:

       Before the `#[cfg(feature = "gateway")]` block, declare:
       ```rust
       #[cfg(feature = "mcp-server")]
       let mut _tools_changed_tx: Option<blufio_mcp_server::notifications::ToolsChangedSender> = None;
       ```

       Then inside the `if config.mcp.enabled` block, after creating the channel, assign:
       ```rust
       _tools_changed_tx = Some(tools_changed_tx);
       ```

       This keeps the sender alive for the lifetime of `run_serve`. The underscore prefix silences the unused-variable warning since no code currently calls `notify()` -- that will happen when skill install events are wired in a future phase.

    3. Update the info! log to include notifications:
       ```rust
       info!(
           memory_resources = memory_store.is_some(),
           notifications = true,
           "MCP HTTP transport enabled at /mcp with resource access",
       );
       ```

    **Gap 2 fix -- ProgressReporter in call_tool (SRVR-14):**

    In `crates/blufio-mcp-server/src/handler.rs`, in the `call_tool` method (lines 396-451):

    1. After the export allowlist check (line 409) and before the tool lookup (line 412), extract the progress token from the request meta and create a ProgressReporter:

       ```rust
       // Extract progress_token from request meta (if present) for progress reporting.
       let progress_token = request
           .meta
           .as_ref()
           .and_then(|m| m.get("progressToken"))
           .and_then(|v| match v {
               serde_json::Value::String(s) => Some(s.clone()),
               serde_json::Value::Number(n) => Some(n.to_string()),
               _ => None,
           });
       let _progress_reporter = crate::notifications::ProgressReporter::new(progress_token);
       ```

       The underscore prefix acknowledges that the reporter is created but not yet passed to `tool.invoke()` (which would require a `BlufioTool` trait change). The reporter is constructed to demonstrate the plumbing path and make the log path reachable. If a progress_token was provided, a debug log could optionally be emitted:

       ```rust
       if _progress_reporter.token().is_some() {
           tracing::debug!(
               tool = tool_name,
               progress_token = ?_progress_reporter.token(),
               "progress reporter created for tool invocation"
           );
       }
       ```

       Place this after the reporter creation but before the tool lookup.

    2. Important: The `_progress_reporter` variable must be in scope during the tool invocation timeout block (lines 433-449) so it could theoretically be used. It naturally will be since it is declared before that block.

    **Test updates in handler.rs:**

    Add a test to `call_tool_direct` helper or add a standalone test that verifies the progress_token extraction logic:

    ```rust
    #[test]
    fn extract_progress_token_from_meta() {
        let mut meta = serde_json::Map::new();
        meta.insert(
            "progressToken".to_string(),
            serde_json::Value::String("tok-42".to_string()),
        );
        let token = meta
            .get("progressToken")
            .and_then(|v| match v {
                serde_json::Value::String(s) => Some(s.clone()),
                serde_json::Value::Number(n) => Some(n.to_string()),
                _ => None,
            });
        assert_eq!(token, Some("tok-42".to_string()));
    }

    #[test]
    fn extract_progress_token_number_from_meta() {
        let mut meta = serde_json::Map::new();
        meta.insert(
            "progressToken".to_string(),
            serde_json::Value::Number(serde_json::Number::from(7)),
        );
        let token = meta
            .get("progressToken")
            .and_then(|v| match v {
                serde_json::Value::String(s) => Some(s.clone()),
                serde_json::Value::Number(n) => Some(n.to_string()),
                _ => None,
            });
        assert_eq!(token, Some("7".to_string()));
    }

    #[test]
    fn extract_progress_token_missing_meta() {
        let meta: Option<serde_json::Map<String, serde_json::Value>> = None;
        let token = meta
            .as_ref()
            .and_then(|m| m.get("progressToken"))
            .and_then(|v| match v {
                serde_json::Value::String(s) => Some(s.clone()),
                serde_json::Value::Number(n) => Some(n.to_string()),
                _ => None,
            });
        assert_eq!(token, None);
    }
    ```
