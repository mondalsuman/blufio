Output: OutputScreener for L4 credential/relay detection, HitlManager for L5 confirmation flow.

    Create `src/output_screen.rs` implementing L4 output screening:

    1. `ScreeningAction` enum: Allow, Redact(String /* redacted content */), Block(String /* reason */), DryRun(String /* what would happen */)

    2. `ScreeningResult` struct: action (ScreeningAction), detection_type (Option<String> — "credential_leak" or "injection_relay"), events (Vec<SecurityEvent>)

    3. Provider-specific credential patterns (CREDENTIAL_PATTERNS static LazyLock<Vec<Regex>>):
       - Anthropic: `sk-ant-[a-zA-Z0-9_\-]{20,}`
       - OpenAI: `sk-[a-zA-Z0-9]{20,}` (but not sk-ant to avoid double match)
       - OpenAI project keys: `sk-proj-[a-zA-Z0-9]{20,}`
       - AWS access key: `AKIA[0-9A-Z]{16}`
       - Database connection strings: `(postgres|mysql|mongodb|redis)://[^\s]+`
       - Bearer tokens: `Bearer\s+[a-zA-Z0-9._\-]{20,}`
       - Generic API keys: `[a-zA-Z0-9_\-]{32,}` — TOO BROAD, skip this. Only match known formats.

    4. Relay detection patterns (RELAY_PATTERNS static LazyLock<RegexSet>):
       - Reuse L1 injection patterns via the classifier. If tool call arguments contain injection-like patterns, that's a relay attempt.
       - Use InjectionClassifier::classify() with source_type="llm_output" for relay detection.

    5. `OutputScreener` struct:
       - config: OutputScreeningConfig
       - dry_run: bool
       - session_failure_count: u32 (mutable counter, per-session)
       - classifier: reference or clone of InjectionClassifier (for relay detection)

       Methods:
       - `new(config: &OutputScreeningConfig, dry_run: bool, classifier: InjectionClassifier) -> Self`
       - `screen_tool_args(&mut self, tool_name: &str, args: &serde_json::Value, correlation_id: &str) -> ScreeningResult`
         * Serialize args to string
         * Check for credential patterns (regex scan)
         * If credential found: redact (replace with [REDACTED]) and return Redact action
         * Check for injection relay (run classifier on args string)
         * If relay detected (score > 0): return Block action
         * Increment failure_count on Redact or Block
         * Generate SecurityEvent::OutputScreening
       - `screen_tool_output(&mut self, tool_name: &str, output: &str, correlation_id: &str) -> ScreeningResult`
         * Same checks as screen_tool_args but on raw string output
         * Used for scanning MCP/WASM results before feeding back to LLM (INJC-06 support)
       - `escalation_triggered(&self) -> bool` — returns true if failure_count >= config.escalation_threshold
       - `reset_session(&mut self)` — reset failure counter (new session)

    6. Credential redaction: replace matched pattern with [REDACTED], preserving surrounding text. Multiple credentials in same content all get redacted.

    7. Update `src/lib.rs` to export: `pub mod output_screen;`

    NOTE: L4 screens tool call ARGUMENTS (buffered before execution), NOT streamed text to user. Per CONTEXT.md: "Stream text to user in real-time, buffer tool call arguments for screening."

  <action>
    Create `src/hitl.rs` implementing L5 human-in-the-loop confirmation:

    1. `HitlDecision` enum:
       - AutoApproved(String /* reason: "safe_tool", "session_trust", "api_bypass", "hitl_disabled" */)
       - PendingConfirmation(HitlRequest)
       - Denied(String /* reason: "timeout", "non-interactive", "max_pending", "user_denied" */)
       - DryRun(String /* what would happen */)

    2. `HitlRequest` struct:
       - correlation_id: String
       - tool_name: String
       - args_summary: String (truncated tool args, not full JSON per CONTEXT.md)
       - risk_level: String ("low", "medium", "high")
       - confirmation_message: String (formatted "Approve [tool_name] with args [summary]? Reply YES/NO")

    3. `ConfirmationChannel` trait (async):
       ```rust
       #[async_trait]
       pub trait ConfirmationChannel: Send + Sync {
           fn supports_confirmation(&self) -> bool;
           async fn send_confirmation(&self, session_id: &str, message: &str) -> Result<(), BlufioError>;
           async fn wait_for_response(&self, session_id: &str, timeout: Duration) -> Result<bool, BlufioError>;
       }
       ```
       This trait will be implemented by channel adapters (Telegram, CLI, etc.) in the wiring plan.

    4. `HitlManager` struct:
       - config: HitlConfig
       - dry_run: bool
       - session_approvals: HashMap<String, HashSet<String>> (session_id -> approved tool types)
       - pending_count: u32

       Methods:
       - `new(config: &HitlConfig, dry_run: bool) -> Self`
       - `check_tool(&mut self, tool_name: &str, tool_args: &serde_json::Value, session_id: &str, source_type: &str, channel_interactive: bool, correlation_id: &str) -> HitlDecision`
         * If HITL disabled: return AutoApproved("hitl_disabled")
         * If dry_run: return DryRun
         * If source_type == "api" or "gateway": return AutoApproved("api_bypass")
         * If tool_name in safe_tools: return AutoApproved("safe_tool")
         * If session_approvals[session_id] contains tool_name: return AutoApproved("session_trust")
         * If !channel_interactive: return Denied("non-interactive") + emit SecurityEvent
         * If pending_count >= max_pending: return Denied("max_pending") + emit SecurityEvent
         * Otherwise: increment pending_count, return PendingConfirmation(request)

       - `resolve_confirmation(&mut self, session_id: &str, tool_name: &str, approved: bool) -> HitlDecision`
         * Decrement pending_count
         * If approved: add tool_name to session_approvals[session_id], return AutoApproved("user_approved")
         * If denied: return Denied("user_denied") + emit SecurityEvent

       - `handle_timeout(&mut self, session_id: &str, tool_name: &str, correlation_id: &str) -> (HitlDecision, SecurityEvent)`
         * Decrement pending_count
         * Return Denied("timeout") + SecurityEvent::HitlPrompt with action="timeout"

       - `clear_session(&mut self, session_id: &str)` — remove session from approvals map + reset count

       - `format_confirmation_message(tool_name: &str, args: &serde_json::Value) -> String`
         * Summarize args (first 200 chars of JSON, truncated with "...")
         * Format: "Approve [tool_name] with args [summary]? Reply YES/NO"

       - `categorize_risk(tool_name: &str) -> &'static str`
         * "high": contains "config", "export", "delete", "erase"
         * "medium": starts with "mcp:" or contains "external"
         * "low": everything else (WASM skills, etc.)

    5. SecurityEvent generation:
       - Denied operations: emit SecurityEvent::HitlPrompt with action="denied", tool_name, risk_level, session_id
       - Timeouts: emit SecurityEvent::HitlPrompt with action="timeout"
       - Approvals: emit SecurityEvent::HitlPrompt with action="approved"

    6. Update `src/lib.rs` to export: `pub mod hitl;`

    NOTE: The actual async confirmation flow (sending message to channel, waiting for reply) will be wired in Plan 04. This plan builds the decision engine and state management. The ConfirmationChannel trait is defined here but implemented by channel adapters later.
