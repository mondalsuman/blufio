**Gap 3 -- GDPR erasure audit event (GDPR-01, AUDT-02):**

In `crates/blufio/src/gdpr_cmd.rs`, in the `cmd_erase` function, after the erasure transaction completes successfully but before the success summary is printed:

1. Use the existing `open_audit_db(config).await?` helper (already used earlier in the function for counting) to get an audit DB connection. If the audit_conn from the earlier call is still in scope, reuse it. Otherwise call `open_audit_db` again.

2. If an audit DB connection is available (Some(conn)):
   - Generate a SHA-256 hash of the user_id for privacy (following GdprEvent pattern from Phase 60 -- `use sha2::{Sha256, Digest}; let hash = hex::encode(Sha256::digest(user.as_bytes()));`)
   - Check which hashing imports are already available in gdpr_cmd.rs or blufio crate. If sha2/hex are not imported, add them (they should be workspace deps from Phase 60).
   - Build the audit entry as a direct SQL INSERT into the audit_entries table following the existing schema. Look at `crates/blufio-audit/src/models.rs` for the AuditEntry schema. The key fields are:
     * `id`: Generate a UUID (uuid::Uuid::new_v4().to_string())
     * `timestamp`: Current ISO 8601 timestamp
     * `event_type`: "gdpr_erasure"
     * `actor`: "cli"
     * `session_id`: None/NULL
     * `details_json`: JSON with `{"user_id_hash": "<hash>", "records_affected": {"messages": N, "sessions": N, "memories": N, ...}}` using the counts from the erasure report
     * `entry_hash`: Compute SHA-256 of (prev_hash || canonical_entry). Look up the last entry's hash first via `SELECT entry_hash FROM audit_entries ORDER BY rowid DESC LIMIT 1`. If no entries exist, use empty string as prev_hash.
     * `prev_hash`: The hash from the previous entry (or empty string if first)
     * `pii_marker`: 1 (contains user reference, even though hashed)
   - Execute the INSERT via `conn.call(move |db| { ... })`.await
   - If the audit write fails, log a warning but do NOT fail the erasure command (audit is best-effort in CLI context, following Phase 54 pattern of non-blocking audit writes).

3. If no audit DB connection (None -- audit.db doesn't exist):
   - Print a yellow warning: "Audit database not found; erasure event not logged. Run `blufio serve` at least once to initialize audit.db."

4. If dry_run is true, skip the audit write entirely (no side effects in dry-run mode).

**Important:** Do NOT use EventBus (not available in CLI context). Do NOT use AuditWriter (requires async runtime with background flushing). Use direct SQL INSERT which is the established pattern for CLI audit access (Phase 54 decision: "CLI audit reads use sync open_connection for direct SQL queries"). Since gdpr_cmd.rs already uses tokio-rusqlite async connections, use `conn.call()` for the INSERT.
